# 日志关键节点输出优化设计

**日期：** 2026-07-05
**状态：** 已批准
**作者：** AI Assistant

## 1. 背景

当前项目已经有基于 Fastify/Pino 的后端日志体系，前端也存在少量调试输出。实际使用中，日志容易被高频明细淹没，尤其是 Prisma 查询日志、Fastify 自动请求日志和前端 SSE chunk 级 `console.log`。

本次优化的目标不是重做日志架构，而是收敛默认输出：只保留能串起一次聊天链路的关键节点，减少日常开发和运行时的噪音。

## 2. 目标

- 后端默认日志只体现关键节点和错误。
- 前端正常聊天流不输出 chunk 级调试日志。
- 不记录用户消息正文、工具完整输入输出、SSE chunk 内容等敏感或高频数据。
- 保留排查一次聊天链路所需的摘要信息。
- 不改变聊天、认证、工具调用等业务行为。

## 3. 非目标

- 不新增完整日志封装层。
- 不引入日志聚合、告警或文件输出。
- 不实现复杂的动态日志策略。
- 不重构无关模块。

## 4. 默认保留的关键节点

### 4.1 后端启动

服务启动成功时保留一条启动日志，继续输出监听地址，便于确认服务可用。

### 4.2 聊天链路摘要

一次聊天请求完成时保留摘要日志，字段以链路定位和规模信息为主：

- `conversationId`
- `toolCallCount`
- `contentLength`
- 必要时保留耗时字段

该日志不包含用户输入正文、助手完整回复或工具完整结果。

### 4.3 LLM 调用摘要

一次 LLM 调用完成或失败时保留摘要日志：

- `model`
- `messageCount`
- `duration`
- 失败时包含错误摘要

### 4.4 工具调用摘要

本次不新增独立的工具调用日志。默认通过聊天链路摘要中的 `toolCallCount` 表达工具调用规模，并通过 `toolNames` 按调用顺序记录具体调用过的工具名。

如果实现中发现已有工具级日志输出完整 `input` 或 `output`，则将其改为摘要字段：

- `toolName`
- 调用状态
- 必要时记录耗时

默认不记录完整 `input` 和 `output`。

### 4.5 错误日志

所有异常路径继续保留 `error` 级别日志，包含定位上下文：

- `requestId`
- `conversationId`（如果存在）
- 标准错误信息

## 5. 默认降噪的内容

### 5.1 Prisma 查询明细

Prisma 每条 query 不再进入默认 `info` 输出。默认日志不展示 SQL 文本和每条查询耗时。

### 5.2 Fastify 自动请求日志

如果 Fastify 自动请求日志导致正常请求刷屏，则关闭或降低默认输出粒度。正常路径依赖业务关键节点日志表达链路状态；错误路径继续由错误处理器记录。

### 5.3 前端 SSE chunk 明细

移除正常流式读取过程中的 chunk 级 `console.log`。前端正常收消息时不输出日志；失败路径可以继续保留必要的错误输出。

## 6. 方案取舍

选用“收敛现有日志点”的方案。

优点：

- 改动小，风险低。
- 保留现有 Fastify/Pino 体系。
- 能直接解决当前输出噪音问题。
- 不引入暂时用不到的配置和抽象。

不选“新增调试开关”的原因：当前需求是默认只体现关键节点，调试开关会增加配置和测试成本，可以在后续确实需要细粒度排查时再补。

不选“重做日志封装层”的原因：当前日志规模不大，新增封装层会扩大改动面，不符合简单优先。

## 7. 测试策略

- 后端测试验证应用仍能生成 `requestId`，错误处理仍返回稳定响应。
- 后端可增加针对日志配置的测试，确认默认不会启用 Prisma query 明细输出。
- 前端测试验证正常聊天流不调用 `console.log` 输出 chunk。
- 现有聊天、认证、工具相关测试继续通过，证明行为未改变。

## 8. 成功标准

- 正常发起一次聊天时，终端只看到少量关键摘要，不再按数据库查询或 SSE chunk 刷屏。
- LLM、Agent 或聊天失败时仍能看到错误和上下文。
- 前端正常流式接收不向 console 输出 chunk 明细。
- 代码改动集中在日志输出相关位置，没有无关重构。
