## Context

当前聊天前端将所有消息内容都按普通字符串输出。assistant 历史消息在 `MessageRow` 中通过纯文本段落渲染，流式中的 `delta` 也在会话页底部按纯文本展示。这种实现虽然简单，但会让模型已经产出的 Markdown 结构丢失表现力，尤其影响标题、列表、链接和代码块的可读性。

这次变更主要发生在前端展示层，但会同时影响三类体验约束：

- 历史消息与流式消息必须保持一致渲染语义
- user 与 assistant 两类消息需要继续区分展示策略
- 引入 Markdown 渲染后需要控制依赖、安全边界与测试覆盖

## Goals / Non-Goals

**Goals:**

- 让 assistant 历史消息按 Markdown 渲染，而不是纯文本直出
- 让流式中的 assistant 临时内容与最终消息遵循同一套 Markdown 展示规则
- 保持 user 消息继续按纯文本展示，避免扩大范围到输入侧富文本
- 将渲染范围限制在安全可控的 Markdown 子集，不启用原始 HTML
- 为常见 Markdown 元素补充基础样式，使代码块、列表和链接具备可读性

**Non-Goals:**

- 不修改后端 SSE 协议、数据库 schema 或共享消息契约
- 不为消息新增 `contentType`、富文本编辑器或多消息格式体系
- 不引入语法高亮、复制代码按钮等增强型代码块功能
- 不调整模型提示词来强制输出 HTML 或其他富文本格式

## Decisions

### 1. 在前端展示层完成 Markdown 渲染

将 Markdown 解释责任放在 `apps/web` 的消息组件，而不是后端或共享契约层。当前消息链路稳定地传递 `content: string`，因此把变化收敛在渲染层可以保持最小改动，并且不影响持久化、流式协议和历史消息接口。

备选方案：

- 在共享契约中增加 `contentType` 字段：语义更完整，但当前收益不足，且会扩大改动面
- 在模型提示词中要求输出 HTML：可控性与安全性较差，不符合现有项目的简单边界

### 2. 仅对 assistant 消息启用 Markdown，user 消息继续纯文本

assistant 输出本身就是面向阅读的结果，适合按 Markdown 展示；user 输入则仍视为原始文本，避免用户输入内容被解释成标题、列表或链接，减少意外渲染与体验歧义。

备选方案：

- 两类消息都渲染 Markdown：实现简单，但会让用户输入的普通符号组合产生意外格式化

### 3. 流式内容也走同一 Markdown 渲染组件

为了满足“历史消息与流式消息展示一致”的目标，流式中的 `delta` 不再单独采用纯文本段落样式，而是复用 assistant 消息的 Markdown 展示能力。这样生成中和完成后的视觉语义一致，只保留“是否已落库”这一个状态差异。

备选方案：

- 流式阶段纯文本、完成后再切 Markdown：实现更稳，但会形成两套视觉规则，与本次目标不符

### 4. 默认禁用原始 HTML，只支持安全可控的 Markdown 子集

渲染器应以 Markdown AST 为核心，不启用原始 HTML 直通能力。支持的重点放在段落、标题、列表、链接、引用、行内代码和代码块这类高频元素上，既满足聊天场景需要，也避免扩大 XSS 风险面。

备选方案：

- 允许 HTML：灵活，但会把安全审查和样式控制复杂度提前引入

### 5. 样式增强限定在消息内容容器内

基础样式只覆盖 Markdown 常见元素，例如：

- 标题层级与间距
- 有序/无序列表缩进
- 链接颜色与换行策略
- 行内代码和代码块背景、圆角、滚动

样式应封装在消息内容容器内，避免影响站点其他文本组件。

## Risks / Trade-offs

- [流式 Markdown 在内容未闭合时可能重排抖动] → 接受轻微抖动，优先换取生成中与生成后的一致语义；实现时保持样式简单，避免额外动画放大抖动
- [新增依赖会增加前端体积与维护面] → 只引入 Markdown 渲染所需最小依赖，不同时叠加高亮、复制等增强功能
- [样式覆盖不足会导致渲染“能用但难看”] → 在测试与验收中明确覆盖列表、代码块、链接、长文本换行和移动端场景
- [assistant 与 user 渲染策略分叉会增加组件判断] → 将判断收敛在消息行组件中，避免传播到数据层和接口层

## Migration Plan

1. 在前端引入 Markdown 渲染能力，并封装为 assistant 消息使用的内容展示组件
2. 将历史 assistant 消息切换到该组件，确认 user 消息仍保持纯文本
3. 将流式 `delta` 展示切换到同一渲染策略
4. 补充消息渲染与流式展示测试，确认旧消息加载与新消息生成都符合预期
5. 如上线后出现渲染异常，可临时回退到纯文本组件，因为本次不涉及持久化或协议迁移

## Open Questions
