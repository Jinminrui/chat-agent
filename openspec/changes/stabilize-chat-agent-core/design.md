## Context

当前仓库的问题不是单点缺陷，而是多条基础链路同时存在断裂与漂移：

- 聊天后端已经有 runtime、provider、tool registry、checkpoint 和 SSE writer，但真实 provider 并未完成工具调用闭环，多轮消息上下文也未稳定接入。
- 前端、后端与 `packages/shared` 之间对用户类型、SSE 事件、错误响应和测试 mock 的理解已经不一致，导致“代码能编、行为未对齐”。
- 认证与会话的后端行为、前端跳转与聊天页入口没有形成统一产品流。
- README、脚本与校验入口对外暴露了可用承诺，但当前仓库缺少最低限度的稳定执行路径。

约束如下：

- 只做让现有承诺能力变得真实可用的修复，不扩展到文件上传、RAG、多模态等新范围。
- 优先复用现有目录与模块边界，不引入新的平台层抽象。
- 需要让修复结果可以通过自动化验证覆盖，而不是只靠手工点通。

## Goals / Non-Goals

**Goals:**

- 让聊天主链路支持稳定的多轮上下文、工具调用循环、checkpoint 恢复与 SSE 错误输出。
- 统一前后端 API 与共享类型契约，消除实现、测试和共享定义之间的漂移。
- 让注册、登录、登出、聊天页访问与会话列表形成闭环用户流。
- 让工程入口与文档能够真实反映项目当前可运行状态。

**Non-Goals:**

- 不新增业务能力，如上传文件、模型切换面板、第三方登录。
- 不进行大规模架构重写，如把 Fastify/Next.js/Prisma 更换为新框架。
- 不为了“未来扩展性”引入当前范围外的抽象层或配置系统。

## Decisions

### 1. 以共享契约为单一事实源，先收敛类型和事件格式

决定优先以 `packages/shared` 中的类型与事件定义为前后端一致的单一事实源，前端 API 层、SSE 解析、页面状态与测试 mock 均围绕该定义收敛。

原因：

- 当前最明显的回归风险来自“同一个概念在三处被重复定义”。
- 先统一契约，后续聊天、认证和测试修复才能避免边修边偏。

备选方案：

- 仅修运行时代码，不先收敛共享类型。放弃，因为这会保留后续反复漂移的根因。

### 2. 聊天路由负责装配历史上下文，runtime 只负责单次执行编排

决定由聊天路由从数据库加载当前会话已有消息，并将“历史消息 + 当前新消息”传入 runtime；runtime 只处理一次执行中的工具循环、工具结果回注和 checkpoint 保存/恢复。

原因：

- 会话归属校验与消息持久化已经在路由层，加载历史消息保持在同一边界最自然。
- runtime 的职责应聚焦在“如何跑一轮 agent”，而不是再去感知 Prisma 或会话查询。

备选方案：

- 让 runtime 自己查询数据库拿历史消息。放弃，因为会把基础设施依赖耦合进执行层。

### 3. checkpoint 恢复采用“已有状态 + 本次新输入”合并，而不是覆盖

决定 checkpoint 只恢复上次工具调用后的执行状态，恢复时必须显式保留本次用户输入，避免旧状态覆盖新消息。

原因：

- 当前最大风险是恢复逻辑把本次请求吃掉，导致多轮对话行为异常且难定位。
- 合并策略可以最小化对现有 runtime 结构的改动。

备选方案：

- 暂时移除 checkpoint。放弃，因为该项目已经持久化了 checkpoint，直接删除会造成更大行为变化。

### 4. SSE 统一为“事件流成功通道 + 事件流错误通道”，不混用 JSON 错误体

决定聊天流一旦开始写出 SSE 头，后续所有执行期错误都通过 `error` 事件结束；仅在写出 SSE 前的认证、参数、归属校验阶段返回普通 HTTP/JSON 错误。

原因：

- 这是最符合当前接口形态的最小修复。
- 前端可按单一模型处理流中断，不再同时兼容“半截 SSE + JSON 500”。

备选方案：

- 把流式接口改成 WebSocket 或非流式。放弃，因为超出当前修复范围。

### 5. provider 保持单 provider 抽象，但补齐真实工具调用协议

决定保留现有 `ChatProvider` 抽象与 OpenAI provider 入口，不新增 provider 层级；仅补齐工具定义传入、工具调用解析和最终答复返回逻辑，并保留无 API key 时的本地 mock 行为。

原因：

- 当前已经有 provider 抽象，问题在“实现缺口”而不是“结构错误”。
- 这是最小成本把 README 中“支持工具调用”变成真实行为的路径。

备选方案：

- 直接砍掉工具调用能力与相关文档。放弃，因为用户明确希望修复而不是降级承诺。

### 6. 认证流以“注册即登录”为准，前端跳转与页面守卫围绕该行为收敛

决定把后端现有注册即写 session 的行为视为目标行为，前端注册成功后直接进入聊天主流程；同时补齐登出动作和聊天页未登录处理。

原因：

- 这是对当前后端最小改动的收敛方向。
- 相比让注册后回登录页，注册即登录能减少一次无意义往返。

备选方案：

- 改后端为注册不登录。放弃，因为会扩大认证链路改动面。

### 7. 工程质量门禁以“可执行最小集合”为目标，而不是一次性建设完整规范

决定本次只要求：

- 根脚本能真实运行前后端测试与必要校验；
- `lint` 不再是假命令；
- README 与实际命令、路径、环境变量保持一致。

原因：

- 用户要的是“全部修复当前问题”，不是顺手搭完整 CI 平台。
- 最小可执行门禁足以支撑本次实现落地。

备选方案：

- 同时引入 ESLint、Prettier、CI 工作流与更严格规范。放弃，因为超出本轮范围。

## Risks / Trade-offs

- [聊天链路改动跨路由、runtime、provider、前端 SSE] → 先统一共享契约与测试，再逐层修聊天执行，减少联动回归。
- [真实工具调用接入后，provider 与 mock 行为可能出现分叉] → 明确 mock 仅用于无密钥开发场景，并为工具调用路径补独立测试。
- [认证流收敛为注册即登录，可能与现有页面文案不一致] → 同步修正前端跳转和文案，避免用户感知混乱。
- [工程命令修正后，可能暴露更多历史测试问题] → 这是预期结果，任务分解中将测试修复视为正式范围，不把它当额外工作。
